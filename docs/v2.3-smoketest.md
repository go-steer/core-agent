Manual UAT for the user-facing features in v2.3:

| PR | What ships | Smoke section |
|----|---|---|
| [#98](https://github.com/go-steer/core-agent/pull/98) | Multi-file instruction loader (`@include` + `AGENTS.d/`) | §1 |

Each section walks an operator-visible reproducer that exercises the
feature end-to-end. If the reproducer behaves as described, the
feature landed.

Out of scope for this smoke (covered by their own UAT processes):

- `examples/gke-parallel-triage/` — needs a real GKE cluster; recipe
  was validated when it landed (PR #95).
- `examples/plan-first/` recipe — exercised throughout §2; not a
  separate section.
- Docs site builds — verify Examples index page renders by running
  `cd docs/site && hugo` if you want; pure markdown so unlikely to
  break.

## Prerequisites

- v2.3 tip on `main` checked out (or `v2.3.0` tag once cut).
- Rebuilt binaries:

  ```bash
  go install ./cmd/core-agent ./cmd/core-agent-tui
  ```

  Verify the version is what you expect:

  ```bash
  core-agent --version
  # expect: core-agent v2.3.0-dev (commit ..., built ...)
  ```

- A model provider credential (`GEMINI_API_KEY` /
  `ANTHROPIC_API_KEY` / Vertex creds).
- A scratch dir so the daemon doesn't inherit an unrelated project's
  `.agents/`:

  ```bash
  mkdir -p /tmp/coreagent-v23-smoke && cd /tmp/coreagent-v23-smoke
  ```

- One terminal for the in-process TUI session is enough for most
  sections. §2.7 needs a second terminal for attach-mode `/replan`.

## §1 — Multi-file instruction loader (PR #98)

### Why we built it

Operators coming from Cursor (`.cursor/rules/*.mdc`), Antigravity
(AGENTS.md with `@include`), or Hermes (`SOUL.md` + governance SOPs)
were forced to flatten everything into a single `AGENTS.md` to use
core-agent. Migration friction killed otherwise-easy adoption. v2.3
adds two composition primitives that map cleanly onto every shape we
checked.

### §1.1 — `@include` directive: happy path

```bash
mkdir -p /tmp/coreagent-v23-smoke/.agents
cd /tmp/coreagent-v23-smoke

# Primary file uses @include to pull in a shared block.
cat > .agents/AGENTS.md <<'EOF'
# Project agent — composed from shared bits.

You are a coding agent helping with the smoke test.

@include base/principles.md

## Project-specific note

The operator is exercising v2.3's multi-file instruction loader.
EOF

mkdir -p .agents/base
cat > .agents/base/principles.md <<'EOF'
## Principles

- Be concise.
- Don't speculate.
- Ask if you're unsure.
EOF
```

Launch the agent:

```bash
core-agent --no-tui -p "Run the /memory slash command. Just list the loaded files; don't comment."
```

**Expected:** the `/memory` output (or the system prompt the model
sees — print it if needed) shows BOTH `.agents/AGENTS.md` AND
`.agents/base/principles.md`. The "Principles" content from the
included file is part of the assembled prompt.

Reverse-check: ask the model to recite the principles. The model
should know about "Be concise / Don't speculate / Ask if you're
unsure" — those came from the included file, not the primary.

### §1.2 — `AGENTS.d/` directory loading

```bash
rm .agents/AGENTS.md
mkdir -p .agents/AGENTS.d

cat > .agents/AGENTS.md <<'EOF'
# Primary AGENTS.md — overlay loaded after this.
EOF

cat > .agents/AGENTS.d/10-tools.md <<'EOF'
## Tools guidance
Prefer `read_file` over `bash cat`.
EOF

cat > .agents/AGENTS.d/20-style.md <<'EOF'
## Style
Output in tight bullets.
EOF

cat > .agents/AGENTS.d/30-workflows.md <<'EOF'
## Workflows
For multi-file changes, always read every relevant file first.
EOF
```

Launch the agent:

```bash
core-agent --no-tui -p "Run /memory and show the loaded files in order."
```

**Expected:** four entries — primary + three `AGENTS.d/*.md` in
lexical order (`10-tools.md` → `20-style.md` → `30-workflows.md`).
Verify the file order matches by asking the model to recite the
first line of each overlay.

### §1.3 — Frontmatter is stripped

```bash
cat > .agents/AGENTS.d/40-with-frontmatter.md <<'EOF'
---
title: Workflow guidance
tags: [internal, do-not-show]
---
## Workflow

Always confirm before deleting files.
EOF
```

```bash
core-agent --no-tui -p "Did your instructions mention the YAML tags 'internal' or 'do-not-show'? Answer yes or no."
```

**Expected:** `no`. The frontmatter block is stripped before the
model sees it — only "## Workflow" + the body is in the prompt.

### §1.4 — Path-escape rejection (fatal in v2.3+)

```bash
cat > .agents/AGENTS.md <<'EOF'
@include ../../etc/passwd
EOF
```

```bash
core-agent --no-tui -p "say hi"; echo "exit: $?"
```

**Expected:** stderr line `core-agent: instruction load: instruction: @include "../../etc/passwd": path escapes scope root (...)` followed by **non-zero exit code (2 / ExitConfigError)**. The loader refuses to follow the `@include` outside the scope root, and v2.3+ treats instruction-load errors as fatal so the agent never runs against a degraded prompt.

Restore the file before continuing:

```bash
cat > .agents/AGENTS.md <<'EOF'
# Primary AGENTS.md — overlay loaded after this.
EOF
```

### §1.5 — Cycle handling via dedup

```bash
cat > .agents/AGENTS.md <<'EOF'
# Primary
@include cycle-a.md
EOF

cat > .agents/cycle-a.md <<'EOF'
A start
@include cycle-b.md
A end
EOF

cat > .agents/cycle-b.md <<'EOF'
B start
@include AGENTS.md
B end
EOF
```

```bash
core-agent --no-tui -p "Ask /memory and list the files in load order."
```

**Expected:** no error, no infinite loop. Each file appears exactly
once. "A start", "A end", "B start", "B end" each show up exactly
once in the prompt (the cycle is broken by the canonical-path
visited-set; second encounter is a silent skip).

### §1.6 — Missing `@include` is a load error (fatal in v2.3+)

```bash
cat > .agents/AGENTS.md <<'EOF'
@include nope.md
EOF
```

```bash
core-agent --no-tui -p "say hi"; echo "exit: $?"
```

**Expected:** stderr line `core-agent: instruction load: instruction: @include "nope.md": file not found (resolved to .../nope.md)` followed by **non-zero exit code (2 / ExitConfigError)**. Operator typos surface immediately rather than silently shrinking the prompt; the agent never runs.

### §1.7 — `.agents/` subdir + project-root both load additively

```bash
rm -rf .agents AGENTS.md AGENTS.d cycle-*.md
mkdir -p .agents AGENTS.d

# Two AGENTS.md files — one in each location.
cat > AGENTS.md <<'EOF'
ROOT-LEVEL-CONTENT
EOF

cat > .agents/AGENTS.md <<'EOF'
SUBDIR-LEVEL-CONTENT
EOF
```

```bash
core-agent --no-tui -p "What two distinctive marker strings appeared in your instructions? List exactly two."
```

**Expected:** the model lists both `ROOT-LEVEL-CONTENT` and `SUBDIR-LEVEL-CONTENT`. Both locations load — additive semantics. The `.agents/AGENTS.md` content appears first in the prompt (preferred location precedence), `<root>/AGENTS.md` second.

Clean up before §2:

```bash
rm -rf .agents AGENTS.md AGENTS.d
```

## §2 — Plan-first gating (PR #100)

### Why we built it

Hermes and others enforce "research → plan → execute" via prose in
AGENTS.md ("you are strictly forbidden from running raw kubectl…").
Frontier models mostly comply; smaller/faster models drift. v2.3
adds gate-level enforcement: until the model calls `record_plan`,
all mutating tools (`write_file`/`edit_file`/`delete_file`/`bash`/
spawn family / MCP tools) deny — at the substrate level, not the
prompt.

### §2.1 — Drop-in recipe + first run

```bash
mkdir -p /tmp/coreagent-v23-smoke
cd /tmp/coreagent-v23-smoke
cp -r $(go env GOPATH)/src/github.com/go-steer/core-agent/examples/plan-first/. ./
# or wherever your core-agent checkout lives; the example dir is examples/plan-first/

ls .agents
# expect: AGENTS.md, config.acceptedits.json.example, config.json, config.yolo.json.example
```

Confirm the default variant is `ask + require_plan_artifact`:

```bash
cat .agents/config.json | grep -E "mode|require_plan"
# expect:
#   "mode": "ask",
#   "require_plan_artifact": true,
```

### §2.2 — Mutating tool denied pre-plan

```bash
core-agent --no-tui -p "Create a file called /tmp/foo.txt with the content hello"
```

**Expected:** the agent attempts `write_file` (or `bash`), the gate
denies with "plan-first mode requires record_plan to be called
before any mutating tool. Call record_plan(plan: <your-markdown-
plan>) first, then retry". The model should recover by calling
`record_plan`.

Check the plan artifact landed:

```bash
ls .agents/plans/
# expect: plan-1.md

cat .agents/plans/plan-1.md
# expect: structured markdown plan from the model
```

### §2.3 — Read tools work pre-plan

```bash
rm -rf .agents/plans  # reset gate state via fresh session

cat > /tmp/coreagent-v23-smoke/sample.txt <<'EOF'
The quick brown fox jumps over the lazy dog.
EOF

core-agent --no-tui -p "Use read_file to read /tmp/coreagent-v23-smoke/sample.txt and tell me what animal it mentions. Don't write anything."
```

**Expected:** the agent reads the file (no plan-first denial; read
tools are exempt) and answers "fox" (or "dog"). No `record_plan`
call.

### §2.4 — Plan in chat scrollback

Run §2.2 again. While the agent's response streams:

**Expected:** the `record_plan` tool call appears as a tool-call
card in chat; the plan content (markdown) is visible to the
operator. After it returns, subsequent `write_file` calls succeed
(per ask-mode prompts).

### §2.5 — Composition: `yolo + require_plan_artifact`

```bash
cp .agents/config.yolo.json.example .agents/config.json
rm -rf .agents/plans

core-agent --no-tui -p "Create a file /tmp/coreagent-v23-smoke/yolo-test.txt with content 'yolo'"
```

**Expected:**
1. First mutating-tool call denies with the plan-first message.
2. Agent calls `record_plan`.
3. After `record_plan` returns, `write_file` (and bash, etc.)
   auto-allow with **no prompts** — that's yolo + plan-first's
   "just tell me the plan, then go" semantics.

Verify the file was written:

```bash
cat /tmp/coreagent-v23-smoke/yolo-test.txt
# expect: yolo
```

### §2.6 — Composition: `acceptEdits + require_plan_artifact`

```bash
cp .agents/config.acceptedits.json.example .agents/config.json
rm -rf .agents/plans
rm -f /tmp/coreagent-v23-smoke/yolo-test.txt

core-agent --no-tui -p "Create /tmp/coreagent-v23-smoke/edits-test.txt with content 'ok' AND then run bash to echo hello"
```

**Expected:**
1. Plan-first denial → `record_plan` → plan written.
2. `write_file` auto-allows (acceptEdits mode after plan).
3. `bash` PROMPTS the operator (acceptEdits doesn't auto-allow
   shell; only file writes).

Approve the bash prompt. Verify both side effects:

```bash
cat /tmp/coreagent-v23-smoke/edits-test.txt
# expect: ok
```

### §2.7 — `/replan` slash command

Restore the default config and start a fresh session:

```bash
cp .agents/config.json .agents/config.json.acceptedits.bak 2>/dev/null
# (or `git checkout` if the example is in a checkout)
# Recreate the default mode=ask variant if needed.

rm -rf .agents/plans
```

Drive a session that gets to a recorded plan, then revoke:

```bash
core-agent --no-tui -p "Plan to add a HEAD endpoint to a hypothetical HTTP server. Just record the plan; don't write any code yet."
```

**Expected:** `plan-1.md` exists in `.agents/plans/`.

Now run `/replan` from the in-process TUI (or via attach mode if
you're set up for it):

In the in-process TUI:
1. Launch `core-agent` (no `-p`, full TUI).
2. Type `/replan reconsider scope` and hit Enter.
3. **Expected:** confirmation message "Plan revoked. Archived to
   .agents/plans/plan-1-revoked.md. The next mutating tool call
   will be denied until the agent calls record_plan again."
4. Verify: `.agents/plans/plan-1.md` no longer exists;
   `.agents/plans/plan-1-revoked.md` does.
5. Ask the agent to write a file → expect plan-first denial → expect
   `record_plan` → expect `plan-2.md` (not plan-1) on disk.

### §2.8 — Subagent inherits parent's plan flag (optional)

Requires the agent to call `spawn_agent`. If you have an example
that fans out subagents, drive it:

1. With plan-first on, the parent records a plan.
2. Parent calls `spawn_agent`.
3. Subagent's first mutating tool call should **succeed** without
   a separate `record_plan` call — the subagent inherits the
   parent's `planRecorded` flag via the shared gate.

Skip if no convenient `spawn_agent` driver; the unit tests cover
this case (`TestPlanFirst_GatesUnknownToolByName`,
`TestPlanFirst_UnblocksAfterMarkPlanRecorded`).

### §2.9 — Cross-restart durability (optional)

The `planRecorded` flag is in-memory; a new daemon process starts
with it false. Recording a plan in one session does NOT carry over
to a new `core-agent` invocation. That's expected for v1; if a
operator wants persistent flag state across restarts, they'd need
to record again. Confirm:

```bash
rm -rf .agents/plans

core-agent --no-tui -p "Record a plan to verify cross-session flag."
ls .agents/plans/  # plan-1.md exists

core-agent --no-tui -p "Now try to write a file."
# Expect: plan-first denial — flag is fresh-process false; the
# previous plan-1.md is irrelevant to gate state.
```

This is **expected** behavior. If operators complain about it,
file-backed `planRecorded` is a v2.4 follow-up (gap #1 from the
plan-first design doc).

## §3 — Smoke pass criteria

The smoke is **clean** when:

- §1.1–§1.6 all behave as described
- §2.1–§2.7 all behave as described
- §2.8 and §2.9 are optional / informational
- No panics, no hung processes, no goroutine leaks visible in the
  daemon logs

If any section fails: file as a release blocker, fix, re-run
affected section, then run the full pass once more before tagging
`v2.3.0`.

## §4 — What this smoke does NOT cover

- **Real-world `examples/kube-platform-agent/`** — not built yet;
  filed in `docs/kube-agents-platform-fit.md`.
- **Plan-first under `agent.RunAutonomous`** — the unit tests
  cover the gate logic; a real autonomous run with plan-first
  would be valuable but isn't a v2.3-release blocker (RunAutonomous
  composes the same gate machinery; behavior is unit-test-verified).
- **Plan-first + MCP tools** — Q1's "gate everything by default"
  resolution means MCP tools are plan-gated. Worth poking with the
  GKE recipe once `examples/kube-platform-agent/` lands; not
  covered here.
- **Loader v2 + Hermes-shape `SOUL.md` layered via `AGENTS.d/`**
  — covered theoretically by `docs/kube-agents-platform-fit.md`;
  needs a real kube-agents-flavored project to exercise. Not a
  v2.3 blocker.
- **Performance** — no benchmarks. Loader and gate pre-check are
  both O(small) per turn; no expected perf regression.
- **core-tui rendering of long `record_plan` tool-call cards** —
  rendering uses today's inline tool-call card (Q5 deferred to
  v2.4 polish per design). 3KB plans may scroll the chat; not a
  blocker.
